02Base de datos — Modelos JPA y esquema PostgreSQL
Diseño de entidades

Hay cuatro entidades principales, cada una mapeada a una tabla PostgreSQL mediante anotaciones JPA estándar. Todos los IDs son UUID v4 (no INTEGER autoincremental), lo que facilita la distribución y evita colisiones entre entornos.

User.java Criterio 3
@Entity @Table(name = "users")
@Data @Builder
public class User {

  @Id @UuidGenerator
  private UUID id;

  @Column(nullable = false, unique = true)
  private String email;

  @Enumerated(EnumType.STRING)
  private Role role;      // trainer | user

  private UUID trainerId; // FK → entrenador asignado
  private String supabaseId; // vínculo con Auth

  @PrePersist
  void onCreate() {
    createdAt = LocalDateTime.now();
  }

  public enum Role { trainer, user }
}

supabaseId: columna única que vincula cada fila de la tabla users del backend con el usuario en Supabase Auth. Se usa en JwtAuthFilter para resolver el JWT → usuario de BD.

@PrePersist: hook de ciclo de vida JPA. No necesita lógica de negocio para rellenar createdAt; Hibernate lo llama automáticamente antes del INSERT.

Routine.java Criterio 3
@Entity @Table(name = "routines")
public class Routine {

  @Column(nullable = false)
  @Enumerated(EnumType.STRING)
  private Category category;
  // 'gym' | 'yoga' | 'pilates'

  @Enumerated(EnumType.STRING)
  private Difficulty difficulty;
  // 'beginner' | 'intermediate' | 'advanced'

  private Boolean isPublic;
  private UUID createdBy; // FK → users.id

  public enum Category
    { gym, yoga, pilates }
  public enum Difficulty
    { beginner, intermediate, advanced }
}

@Enumerated(STRING): los valores se almacenan como texto legible en la BD ('gym', no 0). Así el esquema SQL puede usar CHECK (category IN (...)) y los datos son legibles sin código Java.

isPublic: las rutinas creadas por entrenadores pueden marcarse como públicas para que todos los usuarios las vean.

03Capa de conexión — Spring Boot ↔ PostgreSQL Criterio 3
application.yml — Connection Pool HikariCP
spring:
  datasource:
    url: ${DB_URL}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 5
      minimum-idle: 1
      connection-timeout: 20000
  jpa:
    hibernate.ddl-auto: validate
    properties.hibernate:
      dialect: PostgreSQLDialect

HikariCP mantiene un pool de 5 conexiones reutilizables. En lugar de abrir/cerrar una conexión TCP por cada petición HTTP (costoso), las conexiones ya están abiertas y listas. El límite de 5 encaja con el plan gratuito de Supabase.

ddl-auto: validate: Hibernate verifica que las entidades Java coincidan con el esquema real de la BD, pero nunca lo modifica. El esquema se gestiona con supabase-schema.sql.

Repositorios JPA — Spring Data
// Spring genera la implementación automáticamente
public interface RoutineRepository
    extends JpaRepository<Routine, UUID> {

  // Spring traduce el nombre → SQL WHERE:
  List<Routine> findByIsPublicTrue();

  List<Routine> findByIsPublicTrueAndCategory(
    Routine.Category category);

  List<Routine> findByIsPublicTrueAndDifficulty(
    Routine.Difficulty difficulty);

  List<Routine> findByIsPublicTrueAndCategoryAndDifficulty(
    Routine.Category cat,
    Routine.Difficulty diff);
}

Con Spring Data JPA no hace falta escribir SQL para operaciones estándar. El framework interpreta el nombre del método y genera la consulta: findByIsPublicTrueAndCategory → SELECT * FROM routines WHERE is_public = true AND category = ?.

Patrón upsert en AuthController — sincronización usuario Supabase ↔ BD

Cuando un usuario se autentica en Supabase, la app llama a POST /api/auth/sync. El backend busca al usuario en su propia BD por supabase_id. Si no existe, lo crea; si ya existe, actualiza sus datos. Esto mantiene coherencia entre ambas fuentes de verdad sin duplicados.

// MovoControllers.java — AuthController
User user = userRepo
  .findBySupabaseId(req.supabaseId())   // ¿ya existe este usuario?
  .orElseGet(() -> {                     // no existe → creamos uno nuevo
    User.Role role = User.Role.valueOf(req.role().toLowerCase());
    return User.builder()
      .supabaseId(req.supabaseId())
      .email(req.email())
      .fullName(req.fullName())
      .role(role)
      .build();
  });

user.setFullName(req.fullName());   // actualiza si ya existía
return ResponseEntity.ok(userRepo.save(user)); // INSERT o UPDATE

RoutineController — Lógica de filtrado con 4 ramas

El endpoint GET /api/routines acepta parámetros opcionales category y difficulty. En lugar de concatenar SQL dinámico (vulnerable a SQL injection), se convierten a enums Java y se selecciona la query JPA adecuada según qué parámetros llegaron:

@GetMapping
public ResponseEntity<List<Routine>> list(
    @RequestParam Optional<String> category,
    @RequestParam Optional<String> difficulty) {

  // Conversión segura String → Enum con try/catch
  Routine.Category cat = category.map(s -> {
    try { return Routine.Category.valueOf(s.toLowerCase()); }
    catch (IllegalArgumentException e) { return null; }
  }).orElse(null);

  // Selección de query según combinación de filtros (4 ramas):
  if (cat != null && diff != null)
    return routineRepo.findByIsPublicTrueAndCategoryAndDifficulty(cat, diff);
  else if (cat != null)
    return routineRepo.findByIsPublicTrueAndCategory(cat);
  else if (diff != null)
    return routineRepo.findByIsPublicTrueAndDifficulty(diff);
  else
    return routineRepo.findByIsPublicTrue();
}

La conversión con try/catch es clave: si el usuario manda category=hacking, el valueOf lanza IllegalArgumentException y devolvemos null → la rama sin filtro devuelve todas las rutinas públicas, en lugar de crashear el servidor.

SessionController — Ciclo de vida de un entrenamiento
// 1. Crear sesión (al pulsar "Iniciar entrenamiento")
@PostMapping("/start")
public WorkoutSession start(@RequestBody StartRequest req,
                             @AuthenticationPrincipal User user) {
  var s = WorkoutSession.builder()
    .userId(user.getId())      // CRITERIO 5: del JWT, no del body
    .routineId(req.routineId())
    .build();
  return sessionRepo.save(s);
}

// 2. Marcar como completada (al finalizar el entrenamiento)
@PutMapping("/{id}/complete")
public WorkoutSession complete(@PathVariable UUID id,
                               @RequestBody CompleteRequest req) {
  return sessionRepo.findById(id).map(s -> {
    s.setCompletedAt(LocalDateTime.now());
    s.setDurationMinutes(req.durationMinutes());
    s.setRating(req.rating()); // 1-5
    return sessionRepo.save(s);
  }).orElse(ResponseEntity.notFound().build());
}

// 3. Estadísticas (para ProgressScreen.tsx)
@GetMapping("/stats")
public StatsResponse stats(@AuthenticationPrincipal User user) {
  long total   = sessionRepo.countCompletedByUserId(user.getId());
  int  minutes = sessionRepo.sumDurationByUserId(user.getId());
  return new StatsResponse(total, minutes, 0, new int[7]);
}

El userId siempre viene del JWT, nunca del cuerpo de la petición. Si se permitiera enviarlo en el body, un usuario malintencionado podría registrar sesiones como otro usuario.

05API Externa — Groq AI (Llama 3.3-70B) Criterio 4
GroqAIService.java — Arquitectura del AI Coach

El AI Coach no llama a Groq desde la app móvil. La lógica vive en el backend: el servidor construye un system prompt personalizado con el perfil del usuario obtenido de la BD, añade el historial de conversación, y envía todo a Groq mediante Spring WebClient (cliente HTTP reactivo y no bloqueante).

@Service
public class GroqAIService {

  @Value("${app.groq.api-key}")   // inyectado desde env var
  private String groqApiKey;

  @Value("${app.groq.model}")      // llama-3.3-70b-versatile
  private String groqModel;

  // Historial en memoria por conversationId
  private final Map<String, List<Map<String,String>>> conversations
    = new HashMap<>();

  public ChatResult chat(UUID userId, String message, String convId) {

    // Paso 1: construir contexto desde la BD (nombre, sesiones completadas...)
    String context = buildUserContext(userId);

    // Paso 2: system prompt personalizado con los datos del usuario
    String systemPrompt = """
      Eres MOVO Coach, entrenador virtual experto en fitness, yoga y pilates.
      Hablas en español, eres motivador y cercano.
      Perfil del usuario: %s
      Máximo 150 palabras por respuesta.
      """.formatted(context);

    // Paso 3: limitar historial a 20 mensajes (evitar superar token limit)
    if (history.size() > 20)
      history = history.subList(history.size() - 20, history.size());

    // Paso 4: llamar a Groq con WebClient (no bloqueante)
    var response = WebClient.builder()
      .baseUrl(groqBaseUrl)
      .defaultHeader(HttpHeaders.AUTHORIZATION,
                     "Bearer " + groqApiKey)
      .build()
      .post().uri("/chat/completions")
      .bodyValue(requestBody)
      .retrieve()
      .bodyToMono(JsonNode.class)
      .block();
  }
}

Ventaja de seguridad: la GROQ_API_KEY nunca sale del servidor. Si se llamara a Groq desde la app móvil, cualquiera podría extraerla del código bundle y usarla libremente.

Historial limitado a 20 mensajes: el modelo Llama 3.3 tiene 128K tokens de ventana de contexto, pero enviar conversaciones largas hace las respuestas más lentas y caras. 20 mensajes cubren una conversación completa con buen rendimiento.

Formato de petición a la API Groq (compatible OpenAI)
// POST https://api.groq.com/openai/v1/chat/completions
{
  "model": "llama-3.3-70b-versatile",
  "messages": [
    { "role": "system",    "content": "Eres MOVO Coach..." },
    { "role": "user",      "content": "¿Qué ejercicio me recomiendas?" },
    { "role": "assistant", "content": "Te recomiendo empezar con..." },

El campo messages incluye todo el historial previo de la conversación. El modelo recibe el contexto completo y puede responder de forma coherente recordando lo anterior. La propiedad temperature: 0.7 equilibra creatividad y coherencia factual.

06Seguridad — JWT ES256 + Spring Security Criterio 5
¿Por qué ES256 (ECDSA) en lugar de HS256 (HMAC)?

HS256 (simétrico): la misma clave se usa para firmar y verificar. Si el backend necesita verificar, tiene que conocer la clave privada, que Supabase guarda en sus servidores. Imposible sin compartirla.

ES256 (asimétrico, P-256): Supabase firma con la clave privada (que nunca sale). El backend solo necesita la clave pública para verificar la firma. La clave pública es pública por definición.

// JwtAuthFilter.java — Reconstrucción de la clave pública EC desde JWKS
// Coordenadas del punto en la curva P-256 (obtenidas del JWKS de Supabase)
private static final String JWK_X = "qU87OhwUzO0YpKxSn2OBu_f7H976uFlYCTlzpt6xtYA";
private static final String JWK_Y = "0B7ZiNwBvKL1n7n0D4OGDcfCnX-EJjD1JeX7P6yElpg";

private PublicKey buildEcPublicKey() throws Exception {
  byte[] x = Base64.getUrlDecoder().decode(JWK_X);
  byte[] y = Base64.getUrlDecoder().decode(JWK_Y);

  ECPoint point = new ECPoint(
    new BigInteger(1, x),
    new BigInteger(1, y));

  AlgorithmParameters params = AlgorithmParameters.getInstance("EC");
  params.init(new ECGenParameterSpec("secp256r1")); // curva NIST P-256

  ECPublicKeySpec spec = new ECPublicKeySpec(
    point, params.getParameterSpec(ECParameterSpec.class));

  return KeyFactory.getInstance("EC").generatePublic(spec);
}
doFilterInternal — Validación por petición
@Override
protected void doFilterInternal(HttpServletRequest req, ...) {

  // 1. Leer header Authorization: Bearer <token>
  String auth = req.getHeader("Authorization");
  if (auth == null || !auth.startsWith("Bearer ")) {
    filterChain.doFilter(req, res); // sin token → sigue sin auth
    return;
  }

  String token = auth.substring(7); // descarta "Bearer "

  // 2. Verificar firma ES256 con clave pública de Supabase
  Claims claims = Jwts.parser()
    .verifyWith(buildEcPublicKey()) // ← verifica firma
    .build()
    .parseSignedClaims(token)     // lanza exc. si expirado/inválido
    .getPayload();

  // 3. Extraer supabase_id del claim "sub"
  String supabaseId = claims.getSubject();

  // 4. Cargar User de la BD y asignar rol (ROLE_USER / ROLE_TRAINER)
  userRepository.findBySupabaseId(supabaseId).ifPresent(user -> {
    String authority = "ROLE_" + user.getRole().name().toUpperCase();
    var authToken = new UsernamePasswordAuthenticationToken(
      user, null,
      List.of(new SimpleGrantedAuthority(authority)));
    SecurityContextHolder.getContext().setAuthentication(authToken);
  });
}

Una vez ejecutado el filtro, cualquier método de cualquier controlador puede inyectar el usuario autenticado con @AuthenticationPrincipal User user sin necesidad de leer tokens manualmente.

SecurityConfig — Cadena de filtros HTTP
return http
  .csrf(csrf -> csrf.disable())
  // CSRF desactivado: las apps móviles usan Bearer, no cookies

  .sessionManagement(sm ->
    sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
  // Sin sesión en servidor → cada request lleva su propio JWT

  .authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/auth/**", "/actuator/health")
    .permitAll()          // login/registro: sin JWT
    .anyRequest().authenticated()) // todo lo demás: JWT requerido

  .addFilterBefore(jwtAuthFilter,
    UsernamePasswordAuthenticationFilter.class)
  // JwtAuthFilter se ejecuta ANTES del filtro de usuario/contraseña
  .build();
07Frontend React Native — Stores Zustand + Axios
api.ts — Cliente Axios centralizado

Toda la comunicación con el backend pasa por una única instancia Axios con un interceptor de request que añade automáticamente el JWT de Supabase a cada petición. Los módulos de la app simplemente llaman a routineApi.getPublic() sin preocuparse de tokens.

// Instancia Axios con baseURL y timeout
const api = axios.create({
  baseURL: API_BASE_URL, // 'http://localhost:8080/api'
  timeout: 5000,
  headers: { 'Content-Type': 'application/json' },
});

// CRITERIO 5: Interceptor de request → añade JWT a cada llamada
api.interceptors.request.use(async (config) => {
  const token = await getAccessToken(); // lee de AsyncStorage
  if (token) config.headers.Authorization = `Bearer ${token}`;
  return config;
});

// APIs agrupadas por dominio:
export const routineApi = {
  getPublic: (filters?) => api.get<Routine[]>('/routines', { params: filters }),
  create: (data) => api.post<Routine>('/routines', data),
};

export const aiApi = {
  chat: (message, convId?) =>
    api.post<AIChatResponse>('/ai/chat', { message, conversationId: convId }),

authStore.ts — Zustand + Supabase Auth
// Estado reactivo de autenticación
export const useAuthStore = create<AuthState>((set, get) => ({
  user: null,
  isAuthenticated: false,
  isLoading: true, // splash mientras comprueba sesión

  login: async (email, password) => {
    // 1. Autenticar con Supabase
    const { data } = await supabase.auth
      .signInWithPassword({ email, password });

    // 2. Sincronizar con backend Spring Boot
    await authApi.syncUser({
      supabaseId: data.user.id,
      email: data.user.email,
      role: data.user.user_metadata.role
    });

    set({ user: ..., isAuthenticated: true });
  },

  initialize: async () => {
    // Comprueba AsyncStorage al arrancar
    const { data: { session } } =
      await supabase.auth.getSession();
    set({ isLoading: false,
          isAuthenticated: !!session });
  },
}));

08Flujos completos de extremo a extremo
Flujo 1 — Login y sincronización

LoginScreen.tsx llama a authStore.login(email, password)

Supabase Auth valida credenciales y devuelve un JWT ES256 firmado con la clave privada de Supabase

El JWT se almacena en AsyncStorage (persistencia local). supabase.auth lo gestiona automáticamente

authApi.syncUser() → POST /api/auth/sync con supabaseId, email y rol

AuthController en Spring: busca o crea el usuario en PostgreSQL (upsert) y devuelve la entidad Java

AppNavigator detecta isAuthenticated = true → navega al stack correcto (User o Trainer)

Flujo 2 — Petición autenticada (ej. listar rutinas)

RoutinesScreen.tsx llama a routineApi.getPublic({ category: 'gym' })

Interceptor Axios añade Authorization: Bearer <JWT> automáticamente antes de enviar

JwtAuthFilter en Spring extrae el token, verifica la firma ES256 con la clave pública de Supabase

Extrae el sub (supabaseId), carga el usuario de la BD desde UserRepository, inyecta ROLE_USER en el SecurityContext

RoutineController.list() recibe el filtro, convierte a enum, ejecuta la query JPA apropiada

JSON de rutinas devuelto → Axios lo deserializa → Zustand actualiza el estado → React Native re-renderiza

Flujo 3 — AI Coach

AICoachScreen.tsx llama a aiApi.chat("¿Qué ejercicio me recomiendas?", convId)

Axios añade el JWT → AIController.chat() recibe la petición con el usuario autenticado vía @AuthenticationPrincipal

GroqAIService.buildUserContext(userId) consulta UserRepository (nombre, objetivos) y WorkoutSessionRepository (sesiones recientes)

Construye el system prompt personalizado e incluye el historial de la conversación (hasta 20 mensajes)

WebClient hace POST https://api.groq.com/openai/v1/chat/completions con el modelo llama-3.3-70b-versatile

Groq devuelve la respuesta generada → el servicio la parsea, actualiza el historial → la app muestra el mensaje del AI Coach

Decisiones técnicas destacadas

UUID como PK: permite generar IDs en el cliente sin consultar la BD, facilita la distribución y evita enumerar recursos.

ES256 vs HS256: la clave privada nunca sale de Supabase; el backend solo necesita la clave pública para verificar, lo que es más seguro y escalable.

userId del JWT, nunca del body: impide que un usuario autenticado suplante a otro enviando un userId arbitrario en el cuerpo de la petición.

Groq en backend: la API key de Groq nunca está en el código de la app móvil. Además, el backend puede enriquecer cada petición con el contexto del usuario antes de enviársela al LLM.

Spring Data JPA query methods: sin SQL manual para las queries de rutinas → sin posibilidad de inyección SQL, code más limpio.

Zustand + interceptor Axios: la gestión del token es completamente transparente para los componentes de UI, que simplemente llaman a las funciones de la API.
