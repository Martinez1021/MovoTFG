Justificación del motor de base de datos

PostgreSQLSupabaseRelacional

¿Qué motor usas y por qué?

Uso PostgreSQL como motor de base de datos, alojado en Supabase (PostgreSQL como servicio en la nube).

❓ ¿Por qué elegiste PostgreSQL y no MySQL, MongoDB u otro?
PostgreSQL es el motor relacional más avanzado del mundo open source. Lo elegí por varias razones técnicas concretas:
  • Soporte nativo de UUID: mis entidades usan UUID como clave primaria (uuid_generate_v4()), que PostgreSQL maneja de forma nativa.
  • Array de texto nativo: la tabla user_profiles tiene columnas goals TEXT[] y preferred_types TEXT[] — esto en MySQL requeriría una tabla intermedia.
  • JSONB: las conversaciones de IA se almacenan en columna messages JSONB, que PostgreSQL indexa y consulta eficientemente.
  • Row Level Security (RLS): Supabase se basa en las políticas RLS de PostgreSQL para multitenancy — cada usuario solo ve sus datos, a nivel de base de datos, no solo de aplicación.
  • Integridad referencial: la app tiene relaciones complejas (usuarios → rutinas → ejercicios → sesiones) que se benefician de las FK con CASCADE.
  • Frente a MongoDB: mis datos son estructurados y relacionales. Un modelo documental perdería integridad referencial y complicaría las consultas JOIN que necesito.
❓ ¿Qué es Supabase exactamente?
Supabase es una plataforma BaaS (Backend as a Service) que proporciona:
  1. PostgreSQL gestionado en la nube (AWS)
  2. API de autenticación (GoTrue) — registro, login, JWT
  3. API REST y Realtime autogenerada desde el esquema
  4. Storage para archivos
Es el equivalente open-source de Firebase pero con SQL real.
❓ ¿Tiene alguna limitación?
Sí, soy consciente de las limitaciones:
  • Vendor lock-in parcial: el sistema de auth (GoTrue) y las políticas RLS están acopladas a Supabase, aunque al ser PostgreSQL puro migrar la BBDD sería viable.
  • Plan gratuito: pausa el proyecto tras 7 días de inactividad y tiene límite de 500 MB.
  • Latencia: al estar en AWS us-east-1, desde España hay ~100-150ms de latencia en la primera consulta.

❓ ¿Está normalizada la base de datos?
Sí, está en Tercera Forma Normal (3FN):
  • 1FN: todos los campos son atómicos. Aunque goals es TEXT[], es un tipo nativo de PostgreSQL, no valores separados por comas.
  • 2FN: no hay dependencias parciales. Cada tabla tiene su PK UUID y todos los atributos dependen completamente de ella.
  • 3FN: no hay dependencias transitivas. Los datos del entrenador están en su propia fila de users, no duplicados en user_profiles.
  • Relación N:M entre usuarios y rutinas se resuelve con tabla intermedia user_routines con constraint UNIQUE(user_id, routine_id).
❓ ¿Por qué usas UUID como clave primaria y no un entero autoincremental?
Por tres motivos:
  1. Seguridad: los IDs numéricos son predecibles (/users/1, /users/2...). Un UUID es imposible de adivinar.
  2. Distribución: Supabase genera los UUIDs en el lado del cliente antes de insertar, lo que permite operaciones optimistas sin round-trip a la BBDD.
  3. Multitenancy: los UUIDs son únicos globalmente, no solo dentro de una tabla, lo que facilita migraciones y federación.
❓ ¿Qué tipo de relaciones tienen tus tablas?
  • 1:1 → users ↔ user_profiles (un usuario tiene un único perfil)
  • 1:N → routines → exercises (una rutina tiene varios ejercicios — con ON DELETE CASCADE)
  • 1:N → users (trainer) → users (clients) mediante trainer_id autorreferencial
  • N:M → users ↔ routines a través de user_routines
  • 1:N → users → workout_sessions

📱 App móvil (React Native) │ ├──► Supabase JS Client ──► PostgreSQL (directo, para auth/datos) │ (supabase.ts) │ └──► Axios ──► Spring Boot Backend ──► HikariCP ──► PostgreSQL (api.ts) (puerto 8080) (pool) (para lógica IA)
❓ ¿Cómo se conecta el backend Java a PostgreSQL?
El backend usa Spring Data JPA con HikariCP como pool de conexiones:
  • HikariCP mantiene un pool de hasta 5 conexiones abiertas en paralelo, evitando el overhead de abrir/cerrar por cada petición.
  • Spring Data JPA abstrae las queries SQL con repositorios: UserRepository extends JpaRepository
  • Hibernate es el ORM que mapea objetos Java ↔ filas SQL
  • ddl-auto: none — Hibernate NO modifica el esquema en producción, solo leo/escribo.
❓ ¿Cómo se conecta la app móvil a la BBDD?
La app usa el cliente oficial de Supabase para JavaScript:
  • Se conecta vía HTTPS/REST al endpoint de Supabase (PostgREST)
  • La sesión se persiste en AsyncStorage (almacenamiento local del móvil)
  • El token JWT se refresca automáticamente (autoRefreshToken: true)
  • Accede directamente a tablas como supabase.from('routines').select('*')
❓ ¿Qué es un pool de conexiones y por qué lo usas?
Sin pool: cada petición HTTP abre una conexión TCP a PostgreSQL (~50ms) y la cierra al terminar. Con 100 peticiones/segundo = 5000 conexiones/segundo abriendo y cerrando.

HikariCP mantiene un conjunto de conexiones ya abiertas y las reutiliza. Es el pool de conexiones más rápido para Java. Con maximum-pool-size: 5 tenemos 5 conexiones permanentes que atienden todas las peticiones.

❓ Explícame el flujo de login completo con las APIs
  1. El usuario pulsa "Iniciar sesión" → la app llama a Supabase Auth API (supabase.auth.signInWithPassword())
  2. Supabase valida credenciales en PostgreSQL y devuelve un JWT firmado con ES256 (algoritmo de curva elíptica)
  3. La app guarda el JWT en AsyncStorage y lo incluye en todas las peticiones al backend: Authorization: Bearer <JWT>
  4. El backend Spring Boot tiene un filtro (JwtAuthFilter) que valida el JWT usando la clave pública EC de Supabase
  5. Si es válido, busca el usuario en la BBDD por supabase_id y lo inyecta como @AuthenticationPrincipal
  6. En paralelo, la app llama a POST /api/auth/sync para sincronizar el usuario en la BBDD del backend Java
❓ ¿Cómo funciona la integración con la IA?
El coach IA usa el modelo Llama 3.3-70B de Meta a través de la API de Groq (que es mucho más rápida que OpenAI):
  1. El usuario escribe un mensaje en la pantalla de Coach IA
  2. La app envía POST /api/ai/chat al backend Java con el mensaje y el conversation_id
  3. El backend construye un prompt de sistema personalizado con el perfil real del usuario (peso, altura, objetivos, historial de sesiones)
  4. Llama a POST https://api.groq.com/openai/v1/chat/completions con el historial de conversación
  5. Devuelve la respuesta a la app con el texto y los tokens usados
El historial se mantiene en memoria (últimos 20 mensajes) para que la IA tenga contexto.
❓ ¿Qué es REST? ¿Por qué no usaste GraphQL o WebSockets?
REST (Representational State Transfer) es un estilo arquitectónico donde cada recurso tiene una URL y se opera con verbos HTTP (GET, POST, PUT, DELETE).

Usé REST porque:
  • Es el estándar más conocido y tiene amplio tooling (Swagger, Postman...)
  • Spring Boot tiene soporte nativo con @RestController
  • Los datos de la app son principalmente CRUD, no necesito subscripciones en tiempo real (para los mensajes podría añadir Supabase Realtime en una mejora futura)

Seguridad en base de datos y APIs

JWT ES256Row Level SecuritySpring SecurityHTTPSSin contraseñas en código

Capas de seguridad implementadas

CAPA 1 — Transporte: HTTPS/TLS en todas las comunicaciones CAPA 2 — Auth: JWT firmado ES256 (curva elíptica P-256) CAPA 3 — Backend: Spring Security filtra TODA petición autenticada CAPA 4 — BBDD: Row Level Security — cada usuario solo ve SUS datos CAPA 5 — Secrets: Variables de entorno, nunca hardcodeados en git
❓ ¿Cómo funciona la autenticación con JWT?
JWT (JSON Web Token) es un token firmado digitalmente que contiene la identidad del usuario:
  • Header: algoritmo (ES256) y tipo
  • Payload: sub (UUID del usuario), email, role, exp (expiración)
  • Signature: firmado por Supabase con su clave privada EC P-256
El backend valida la firma usando la clave pública EC de Supabase (coordenadas X,Y del punto en la curva). Si alguien modifica el payload, la firma no coincide → petición rechazada.

Uso ES256 (ECDSA) en lugar de HS256 porque es asimétrico: Supabase firma con la privada, el backend verifica con la pública. Más seguro para arquitectura distribuida.
❓ ¿Qué es Row Level Security y cómo lo usas?
RLS es una característica de PostgreSQL que permite definir políticas de acceso a nivel de fila, no solo de tabla. Aunque alguien consiga acceder a la BBDD, solo verá sus propios datos.

Ejemplo real en mi app: aunque la tabla workout_sessions tiene sesiones de todos los usuarios, la política RLS hace que cada consulta solo devuelva las filas donde user_id coincide con el usuario autenticado. Esto ocurre dentro de PostgreSQL, no en el código de la aplicación.
❓ ¿Cómo proteges los endpoints del backend?
Con Spring Security + filtro JWT personalizado:
  1. Todos los endpoints /api/** requieren autenticación excepto /api/auth/**
  2. El JwtAuthFilter intercepta CADA petición, extrae el token del header Authorization: Bearer ...
  3. Valida la firma ES256, extrae el supabaseId, busca el usuario en la BBDD
  4. Inyecta el usuario en SecurityContextHolder para que los controllers lo reciban via @AuthenticationPrincipal
  5. Si el JWT es inválido o está expirado, el filtro deja pasar la petición sin autenticar → Spring Security devuelve 401
❓ ¿Guardas contraseñas en el código o en git?
No. Las contraseñas y secretos se gestionan mediante variables de entorno:
  • La BBDD URL, usuario y contraseña están en application.yml con placeholders: ${SUPABASE_DB_PASSWORD}
  • La API key de Groq: ${GROQ_API_KEY}
  • El secreto JWT: ${JWT_SECRET}
  • En local se usan valores por defecto para desarrollo, pero en producción se inyectan como env vars del servidor
  • Las contraseñas de los usuarios las almacena Supabase hasheadas con bcrypt — mi aplicación nunca ve ni almacena contraseñas en texto plano
❓ ¿Qué es CORS y cómo lo configuras?
CORS (Cross-Origin Resource Sharing) es un mecanismo de seguridad del navegador que bloquea peticiones a dominios distintos al origen. En mi backend Spring Boot, configuro CORS para permitir peticiones desde la app móvil (Expo) y denegar el resto. En producción debería restringirse al dominio real en lugar de *.
