Criterio 1

Justificación de PostgreSQL

Se eligió PostgreSQL (vía Supabase) por sus características avanzadas que ninguna alternativa ofrece de forma nativa:

🔑 UUID como Clave Primaria

  • Generado con uuid_generate_v4()
  • Imposible adivinar IDs de otros usuarios
  • Sin colisiones en entornos distribuidos
  • Seguridad por diseño vs. enteros secuenciales
supabase-schema.sql id UUID PRIMARY KEY DEFAULT uuid_generate_v4()

📦 TEXT[] — Arrays Nativos

  • Almacena listas de strings directamente
  • Evita tabla intermedia para goals y preferred_types
  • Único en PostgreSQL — MySQL/SQLite no lo soportan
supabase-schema.sql goals TEXT[] DEFAULT '{}', preferred_types TEXT[] DEFAULT '{}' -- ej: {'gym','yoga','pilates'}

🧩 JSONB — Historial de Chat IA

  • Guarda todo el historial en un solo campo
  • Búsqueda eficiente con operadores << y @>
  • Sin JSONB necesitaríamos tabla ai_messages con joins costosos
supabase-schema.sql messages JSONB DEFAULT '[]' -- [{role:"user", content:"..."}, ...]

🕐 TIMESTAMPTZ + CHECK Constraints

  • TIMESTAMPTZ: timestamps con zona horaria (usuarios globales)
  • CHECK: validación a nivel de BD, última línea de defensa
  • Si la app envía un valor inválido, la BD lo rechaza
supabase-schema.sql rating INT CHECK (rating BETWEEN 1 AND 5) role TEXT CHECK (role IN ('trainer', 'user'))

🛡️ Row Level Security (RLS)

  • Seguridad a nivel de fila nativa en PostgreSQL
  • Cada usuario solo puede ver/modificar sus propias filas
  • Imposible en MySQL, SQLite o MongoDB de forma nativa
  • auth.uid() = UUID del usuario autenticado con JWT
Criterio 2

Diseño de la Base de Datos

8 tablas en 3FN — sin redundancia, cada campo depende únicamente de la clave primaria. Relaciones 1:1, 1:N y N:M implementadas correctamente con claves foráneas y constraints.
# Tabla Relación FK / Constraint Justificación
1 users Auto-referencia trainer_id → users(id) SET NULL Un User puede ser cliente de otro User (entrenador)
2 user_profiles 1:1 con users UNIQUE(user_id) + CASCADE Datos físicos separados → 3FN (distintos atributos)
3 routines 1:N con exercises created_by → users (nullable) NULL = rutina del sistema; FK a creador entrenador
4 exercises N:1 con routines routine_id CASCADE Al borrar rutina se borran sus ejercicios automáticamente
5 user_routines N:M UNIQUE(user_id, routine_id) Tabla de unión. Usuario ↔ Rutina con restricción duplicados
6 workout_sessions N:1 con users user_id CASCADE Historial de entrenamientos. completed_at NULL = en progreso
7 ai_conversations 1:1 con users CASCADE + JSONB Historial chat IA en JSONB. Sin tabla mensajes separada
8 trainer_messages 2 FKs a users trainer_id + user_id → users Mensajería entrenador ↔ cliente, ambos son filas de users

🔗 CASCADE vs SET NULL

ON DELETE CASCADE: al borrar un usuario, se borran todos sus perfiles, sesiones, conversaciones → integridad referencial total.

ON DELETE SET NULL: al borrar un entrenador, sus clientes no se borran — solo pierden la referencia al entrenador (trainer_id = NULL).

🎯 Enums Java ↔ CHECK SQL

Los enums Java Category y Difficulty en Routine.java están sincronizados con los CHECK constraints del SQL.

@Enumerated(EnumType.STRING) private Category category; // → CHECK(category IN ('gym','yoga','pilates'))
Criterio 3

Conexión a la Base de Datos

⚡ HikariCP — Pool de Conexiones

Abrir una conexión TCP a PostgreSQL cuesta ~50-100ms. Sin pool, cada petición HTTP abriría y cerraría su propia conexión.

HikariCP mantiene 5 conexiones abiertas y listas para reutilizar. Límite de 5 = plan gratuito Supabase.

application.yml hikari: maximum-pool-size: 5 connection-timeout: 30000 initialization-fail-timeout: -1

🗂️ JPA / Hibernate ORM

Mapea clases Java ↔ tablas PostgreSQL sin escribir SQL manual. @Entity, @Id, @Column generan las queries CRUD automáticamente.

Routine.java @Entity @Table(name = "routines") public class Routine { @Id @UuidGenerator private UUID id; @Enumerated(EnumType.STRING) private Category category;

📱 React Native — Supabase JS Client

La app accede directamente a Supabase para auth y operaciones de BD en tiempo real. El JWT se persiste en AsyncStorage.

supabase.ts export const supabase = createClient( supabaseUrl, supabaseAnonKey, { auth: { storage: AsyncStorage, autoRefreshToken: true, persistSession: true, detectSessionInUrl: false } } );

🔄 Backend — Repositorios JPA

4 repositorios extienden JpaRepository para CRUD automático + queries personalizadas.

GroqAIService.java // CRITERIO 3: consultas JPA para // personalizar el contexto del AI private final UserRepository userRepository; private final WorkoutSessionRepository sessionRepository;
ddl-auto: none — Hibernate nunca modifica el esquema de la BD. El esquema se gestiona manualmente con supabase-schema.sql. Esto evita pérdidas accidentales de datos en producción.

🔑 ES256 vs HS256

ES256 (asimétrico): Supabase firma con clave privada. El backend solo tiene la clave pública → imposible falsificar tokens aunque hackeen el backend.

HS256 (simétrico): ambos lados necesitan la misma clave. Si hackean el backend, pueden falsificar tokens.

JwtAuthFilter.java // Clave pública EC P-256 de Supabase private static final String JWK_X = "qU87OhwUzO0YpKxSn2OBu_f7H976uFlYCTlzpt6xtYA"; private static final String JWK_Y = "0B7ZiNwBvKL1n7n0D4OGDcfCnX-EJjD1JeX7P6yElpg";

🚫 STATELESS + CSRF deshabilitado

STATELESS: el servidor no guarda ninguna sesión. Cada request lleva su JWT. Escalable horizontalmente.

CSRF deshabilitado: CSRF ataca apps con cookies de sesión. Como usamos tokens Bearer en el header, no hay cookie que robar.

SecurityConfig.java .csrf(csrf -> csrf.disable()) .sessionManagement(sm -> sm .sessionCreationPolicy( SessionCreationPolicy.STATELESS))

🌍 Variables de Entorno

Ningún secreto está hardcodeado en el código del repositorio. Todos se cargan desde variables de entorno:

application.yml api-key: ${GROQ_API_KEY} password: ${SUPABASE_DB_PASSWORD} jwt-secret: ${SUPABASE_JWT_SECRET} # En producción: Railway/Render # en local: application-local.yml # (en .gitignore)

🛡️ Row Level Security

Políticas RLS en PostgreSQL garantizan que cada usuario solo accede a sus propias filas, independientemente de quién haga la petición:

supabase-schema.sql CREATE POLICY "sessions_own" ON workout_sessions USING (user_id IN ( SELECT id FROM users WHERE supabase_id = auth.uid()::text ));
